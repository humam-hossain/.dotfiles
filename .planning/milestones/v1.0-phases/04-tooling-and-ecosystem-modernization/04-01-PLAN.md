  <action>
    Rewrite `.config/nvim/lua/plugins/lsp.lua` so the active implementation uses Neovim `0.11+` native LSP registration per D-01, D-02, and D-03. Remove the `vim.fn.has("nvim-0.11")` compatibility branch and stop using `require("lspconfig")[server].setup(...)` as the primary setup path. The replacement structure must be explicit and boring:

    1. Keep a single plugin spec rooted at `neovim/nvim-lspconfig`, with Mason, mason-lspconfig, mason-tool-installer, fidget, and blink.cmp as dependencies.
    2. Inside the module, split responsibility into local helpers for:
       - diagnostics configuration
       - `LspAttach` / `LspDetach` behavior
       - server definition table
       - Mason package ensure list
       - enable bridge that calls `vim.lsp.config(server_name, server_opts)` followed by `vim.lsp.enable(server_name)`
    3. Preserve `attach.apply_lsp(event.buf)` so user-facing mappings remain centralized per D-16.
    4. Keep blink capability extension, but rename the misspelled `original_capabilites` local and make the merge path clear.
    5. Separate Mason package names from LSP server IDs. The plan must not keep one mixed list where server IDs and formatter package IDs are indistinguishable.
    6. Keep Mason-first provisioning per D-04 and D-05, but do not add runtime assumptions that binaries only exist under Mason paths. Runtime should continue working when tools are installed system-wide per D-06.
    7. Keep inlay hints, document highlights, diagnostics UI, and `fidget.nvim` support unless a given capability is unsupported by the current client.

    Do not scatter new keymaps into the plugin spec. Do not create OS-specific branches beyond guarded runtime checks already required by shared Linux/Windows support per D-14 and D-15.
  </action>
