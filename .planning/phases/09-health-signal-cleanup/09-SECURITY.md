# Phase 09 — Security

> Per-phase security contract: threat register, accepted risks, and audit trail.

---

## Trust Boundaries

| Boundary | Description | Data Crossing |
|----------|-------------|---------------|
| validator → Neovim health engine | Repo automation reads provider output generated by Neovim and plugins, which may include environment-only failures | Plain-text health check output (non-sensitive) |
| tmux/terminal → Neovim keymaps | Terminal and tmux key delivery can block mappings before repo code executes | Key events (non-sensitive) |
| repo open helper → host opener | `vim.ui.open()` and `xdg-open` depend on host OS state outside repo control | Local file paths (non-sensitive) |
| repo health provider → Neovim health UI | Repo code emits severity strings and guidance that users trust to decide whether action is required | Diagnostic strings (non-sensitive) |
| shared probe module → validator and provider | Drift between bash and Lua classification would make health results inconsistent | Tool/plugin metadata (non-sensitive) |
