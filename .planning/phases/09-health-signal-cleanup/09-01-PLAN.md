<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| validator -> Neovim health engine | Repo automation reads provider output generated by Neovim and plugins, which may include environment-only failures |
| tmux/terminal -> Neovim keymaps | Terminal and tmux key delivery can block mappings before repo code executes |
| repo open helper -> host opener | `vim.ui.open()` and `xdg-open` depend on host OS state outside repo control |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-01 | R | `scripts/nvim-validate.sh` | mitigate | Capture the full health buffer to `checkhealth.txt` so audit evidence is inspectable and not reduced to console progress noise. |
| T-09-02 | I | `scripts/nvim-validate.sh health` | mitigate | Fail only on required tools `git` and `rg`, while preserving optional tool warnings so health output remains actionable instead of noisy. |
| T-09-03 | D | `.config/.tmux.conf` | mitigate | Add the tmux companion bindings and require human verification so pane crossing is proven rather than assumed from Neovim-side mapping ownership. |
| T-09-04 | T | `.config/nvim/lua/core/keymaps/registry.lua`, `.config/nvim/lua/core/open.lua` | mitigate | Investigate BUG-020 in the mandated order and change the mapping only when evidence proves terminal delivery failure or another concrete repo defect. |
</threat_model>
